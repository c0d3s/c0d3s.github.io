staying classy


The concept of a Class is integral to the workings of an object-oriented programming language like Ruby. It allows us to create objects with certain shared characteristics (ie methods and variables). Think of a Class like a blueprint: it has contains the outlines for what qualities an object will have. When we build, or initialize an instance of our Class, we fill in the details.


Let's say we have a Car Class. With any mass produced car, a certain formula is followed so that all makes and models have some shared core attributes. Every Honda Civic is going to have four wheels, a chassis, an engine, etc. But there's also optional features to consider. Do we want four doors or two? A manual transmission or automatic? How about a sunroof? Let's translate this scenario to code.

class Car
  def initialize(doors, transmission, sunroof = false)
    raise ArgumentError.new("Must select 2 or 4 doors") unless doors == 2 || doors = 4
    raise ArgumentError.new("Must select 'manual' or 'automatic' transmission") unless transmission == "manual" || transmission == "automatic"
    @doors = doors
    @transmission = transmission
    @sunroof = sunroof
  end
end

We start defining our Car class by making an initialize method. This allows us to create a new instance of Car and pass the arguments that specify the number of doors, the type of transmission, and whether there is a sunroof; note: sunroof is defaulted to false so that if we do not pass a third argument, the car will not, by default, include a sunroof. The next two lines are somewhat self explanatory -- they raise arguments and disallow the creation of an instance of Car that does not have either 2 or 4 doors, or has something other than an automatic or manual transmission. But what about those variables with the @ in front of them?


These are instance variables! They are vital to our the smooth operation of the other methods we will be creating in our Car class. Instance variables are simply variables that can be used in all our instance methods -- ie methods that can we can call on any instance of car. We certainly want to be able to do more with our instance of Car than simply create it. Let's add a couple more methods to clarify these definitions.

class Car
  def initialize(doors, transmission, sunroof = false)
    raise ArgumentError.new("Must select 2 or 4 doors") unless doors == 2 || doors = 4
    raise ArgumentError.new("Must select 'manual' or 'automatic' transmission") unless transmission == "manual" || transmission == "automatic"
    @doors = doors
    @transmission = transmission
    @sunroof = sunroof
  end

  def get_style
    @doors = 2 ? style = "coupe" : style = "sedan"
    return "This car is a #{style}."
  end

  def get_mpg
    case @doors
    when 2
      @transmission == "manual" ? mpg = [29, 36] : mpg = [27, 34]
    when 4
      @transmission == "manual" ? mpg = [28, 35] : mpg =[26, 34]
    end
    return "The mpg of this car is #{mpg[0]} city and #{mpg[1]} highway."
  end

  def sunroof?
    @sunroof == true ? result = "This car has a sunroof" : result = "This car does not have a sunroof"
    return result
end

Here, inside our Class definition of Car once again, we define instance methods to get the style of our car (coupe or sedan), see the mpg of our car (based upon the number of doors and transmission), and check if our car has a sunroof. You will notice each one of these methods uses the aforementioned instance variables to produce a value and return a string. What would happen if we didn't use instance variables? Well, these methods would simply not function. Our Car#get_style method would not be able to see how many doors our car has, since the doors variable originates in the initialize function. Similarly Car#get_mpg would not be able to see the number of doors we have selected, nor would Car#sunroof? be able to see whether an instance of Car has a sunroof. Because in Car#initialize we set the parameters equal to instance variables, our program works, as the arguments that are passed to Car#initialize when we create a new instance will be visible to the other methods in our Car Class. Notice too that in our Car#get_mpg instance method we use a local variable (not an instance variable) for style. This is because style is only accessed within get_style, making an instance variable unnecssary. Let's demonstrate how this all works using a printout of our terminal from irb.


irb(main):001:0> car1 = Car.new(2, "manual", true)
=> #Car:0x007ff39398de18 @doors=2, @transmission="manual", @sunroof=true>
irb(main):002:0> car1.get_style
=> "This car is a coupe."
irb(main):003:0> car1.get_mpg
=> "The mpg of this car is 29 city and 36 highway."
irb(main):004:0> car1.sunroof?
=> "This car has a sunroof"

We start by creating a new instance of Car called car1 with 2 doors, a manual transmission, and a sunroof. Now we can test our instance methods. We call get_style on car one, and sure enough, we are informed our car is a coupe as a result of our 2 doors. We then test our mpg, which takes into account both our number of doors and transmission. By calling this method, we see car1 has a fuel economy of 29 city and 36 highway. Finally, we check on our sunroof. Calling sunroof? on car1 returns the affirmative. Was this an overly simplistic, contrived example used to demonstrate a point? Perhaps. But knowing what we now know about Classes will help us build objects more effectively.
